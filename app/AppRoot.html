What is?

Modular Cube is a lightweight frontend framework developed around modern web standards. It provides a minimal yet expressive system for building dynamic user interfaces, relying solely on native browser features such as Custom Elements, HTML Templates, ES Modules, and the Shadow DOM. It does not introduce virtual DOMs, transpilation layers, or non-standard syntax. Modular Cube allows developers to create user interface components using familiar HTML and JavaScript, while enabling scoped rendering, declarative structure, and encapsulated behaviors without dependencies or build steps.

It is designed for clarity, efficiency, and sustainability in frontend development.

Modular Cube is built for developers who value:

  • Direct control over the DOM with no abstraction layers.
  • Declarative authoring of behavior and structure using standard HTML and JavaScript.
  • Encapsulation and modularity through native web component APIs.
  • Transparent and predictable execution with no hidden reactivity engines or compile-time logic.

By eliminating external dependencies and working entirely within browser-supported technologies, Modular Cube ensures that what you write is exactly what runs in the browser. This encourages long-term maintainability and lowers the barrier to understanding the codebase, both for teams and individual developers.

Guiding Principles

Native First

All features are built on top of native browser capabilities. Rather than abstract or replace them, Modular Cube composes and extends the platform‚Äôs existing strengths: Custom Elements, templates, DOM events, and modules. This guarantees compatibility, avoids overengineering, and leverages the performance optimizations already present in browsers.

Declarative by Design

User interfaces are described in terms of structure and state, rather than procedural updates. Template interpolation, conditional rendering, iteration, and event binding are all expressed directly in HTML, using a consistent and transparent syntax. The framework interprets this markup declaratively, keeping the intent close to the implementation.

Minimal Assumptions

No runtime behavior is hidden. What is defined in a component is what is executed. The framework avoids hidden proxies, side effects, or complex lifecycle dependencies. This results in easier debugging, reduced surprises, and more portable code.

Simplicity Over Abstraction

Instead of introducing new syntax or conceptual layers, Modular Cube encourages developers to build with the browser's primitives. Each feature is lightweight, easy to reason about, and aligned with native web semantics.

Architecture Based on Web Standards

Custom Elements

Encapsulate behavior, state, and rendering logic into self-contained components using JavaScript classes. Each element registers with the browser as a native HTML tag.

HTML Template Element

Templates define reusable blocks of HTML that are parsed but not rendered immediately. They serve as the basis for dynamic rendering and runtime instantiation.

ES Modules

All components and logic are authored as ES modules, allowing clear boundaries, imports, and exports across your codebase using the standard module system.

Shadow DOM

Components use Shadow DOM for true encapsulation of style and structure, isolating internal implementation details from the global DOM tree when needed.

üí° Design Intent

Modular Cube does not aim to create a new way of building the web, it simply leverages the web as it exists today. Its goal is to empower developers to write UI code that is expressive, explicit, and maintainable, while avoiding the hidden complexity often introduced by large abstractions. It makes no assumptions about how your application is structured, but provides lightweight conventions to support common frontend needs: data-binding, conditional logic, event handling, and templating. By using only platform features, Modular Cube ensures forward compatibility and encourages a development style that aligns closely with how browsers operate. It is ideal for projects where performance, clarity, and long-term maintainability are key concerns.

Core Concepts

Lifecycle

The Modular Cube follows the standard Web Component lifecycle, extended with a custom state model that defines how components transition through distinct operational phases. This lifecycle ensures that each component is properly initialized, configured, and rendered in a predictable and modular way.

üìå Lifecycle Overview

The core Web Component lifecycle methods are as follows:

  • constructor()
  • connectedCallback()
  • adoptedCallback()
  • attributeChangedCallback()
  • disconnectedCallback()

Each stage is enhanced with custom lifecycle states to manage the initialization and rendering flow in more detail.

HTMLComponent

constructor: preProcess State

This is the preparatory stage before the component is rendered or attached to the DOM. It is responsible for receiving and interpreting the component's configuration. The constructor expects a mandatory configuration object with the following structure.

The preProcess state is responsible for validating and loading this configuration before the component begins processing or rendering.

connectedCallback: toProcess, atProcess, inProcess States

Once the component is attached to the DOM, it transitions through three distinct internal states within connectedCallback.

toProcess: This state indicates that the component has been attached to the DOM and is now entering the processing lifecycle. At this stage, it is recognized as active, but no rendering or data processing has begun. It serves as a preparation phase where the component may validate its configuration or initiate resource loading. This state ensures that all necessary prerequisites are in place before rendering starts.

atProcess: At this stage, the component is now in the rendering stage. Required resources have been loaded, and dependencies resolved. The component has reached the point where rendering logic is about to be executed. This state represents the transition from preparation to visual output and is the final checkpoint before the component becomes visible or interactive.

inProcess: This state occurs after the component has completed its rendering phase. The DOM is now fully available for manipulation, and all initial behaviors are active. At this point, logic that depends on the rendered structure, such as querying the DOM with querySelector, setting focus, or attaching runtime event listeners can be safely executed.

Component Setup Guide

The system follows a modular component architecture based on Web Components and the Shadow DOM. Each component is defined across a consistent set of files that separate structure, style, behavior, and configuration. This separation enhances maintainability, reusability, and encapsulation.

File Structure and Naming Convention

Each component consists of four main files, sharing the same base name

For example, a component named AppRoot would be composed of:

  • AppRoot.js
  • AppRoot.html
  • AppRoot.css
  • AppRoot.config.json

All components extend a base class provided by the system: HTMLComponent, imported from the @modular-cube package.

Configuration File (.config.json)

The .config.json file defines metadata, file dependencies, and behavioral flags for each component. It allows the component to be automatically configured and loaded during runtime. Example: AppRoot.config.json

name: Logical name of the component class.

description: Optional documentation string describing the component's purpose.

files.html.path: Path to the component‚Äôs HTML template.

files.css.path: Path to the CSS styles for the component.

behavior.apendToBody: Indicates whether the component should be appended to the body tag automatically at load time.

behavior.lazyLoading: Enables lazy instantiation via IntersectionObserver or a similar mechanism.

Component File (.js)

The JavaScript file defines the actual component class. It imports the configuration JSON file and integrates it into the component lifecycle using the preProcess() method inherited from HTMLComponent.

Application Configuration (app.config.json)

To make a component available to the application, it must be registered in the global application configuration file: app.config.json. This setup allows the framework to dynamically load and define the components at runtime.

tag: The custom element tag name under which the component will be registered (must follow the kebab-case format).

path: Path to the JavaScript module defining the component.

Work with HTML

Variable

Template interpolation enables the embedding of JavaScript expressions directly within HTML content, using the $ {...} syntax. This mechanism allows dynamic rendering of component state and computed values inline within native HTML templates. Interpolation is resolved at render time and evaluated within the current data context (typically the component instance this or the shadow root scope).

üìå Constraints and Best Practices

  • Only valid JavaScript expressions are allowed. These must reference properties or methods available in the current scope.
  • Functions used within expressions must be side-effect-free and return a value.
  • Expressions should remain simple and easy to read. Complex logic or deeply nested expressions should be avoided.

üîç Syntax & Supported Contexts

Expressions are enclosed in $ { expression }:

  • Text content within HTML elements.
  • Attribute values (e.g., class, title, placeholder).
  • Class attribute for dynamic class names.
  • Inline styles via the style attribute

The following examples show proper uses of interpolation for binding values in different parts of the DOM.

Param Tag Replacement

The <param value=""> feature allows direct injection of parameter values into HTML content by specifying dynamic values through component parameters. This mechanism functions similarly to template interpolation but is limited to inserting parameterized values into designated HTML attributes or content areas. Values assigned via <param value=""> are evaluated against the component‚Äôs parameter scope during the initial render

üìå Constraints and Best Practices

  • Each <param> element with a value attribute is replaced with the current value of the corresponding parameter during the initial render.
  • Parameter values are evaluated from the component‚Äôs declared parameter scope.
  • Parameter values injected via should be serializable and renderable as strings suitable for direct HTML insertion.
  • Complex or computed values should be resolved before assignment to parameters.
  • The <param> tag is a placeholder element and does not produce standalone DOM elements after rendering; it is replaced entirely by the parameter‚Äôs value.

üí° Notes

  • Only <param> elements with a value attribute are replaced. Any other usage will be ignored.
  • This interpolation system is part of the framework and not native to the browser. It's optimized for clarity, and Shadow DOM compatibility.

Template interpolation in newFameWork facilitates embedding dynamic data into HTML templates using $ {...} expressions. Evaluation occurs initially during render and later only when explicitly requested, enabling predictable and efficient updates without automatic reactive tracking.

The <param value=""> feature in newFameWork provides a mechanism for injecting component parameter values directly into HTML content. It complements template interpolation by enabling straightforward pass-through of parameter values, evaluated initially at render and updated explicitly on demand. This feature maintains predictable rendering behavior and minimizes unnecessary DOM manipulations by leveraging explicit update triggers.

if, else if, and else

The if directive in Modular Cube provides conditional rendering in HTML templates using declarative expressions. It enables developers to control whether a given element and its children should be rendered based on the evaluation of a boolean condition.

üìå Feature supports

  • These directives follow a familiar control-flow structure: they are evaluated in order, and only the first matching condition will be rendered.
  • else if must directly follow a if or another else if.
  • Only one else block is allowed per if chain.

üí° Notes

In JavaScript, you can absolutely use if, else if, else if without a final else. The else block is completely optional.

  • ‚úÖ You can have just if
  • ‚úÖ You can have if...else
  • ‚úÖ You can have if...else if...else
  • ‚úÖ You can have if...else if...else if (with or without else)

The if directive allows declarative, structured control of what elements are rendered based on logical conditions. Support for else-if, else, and nested conditionals provides flexibility while maintaining clear and predictable rendering behavior. The use of native HTML syntax and controlled evaluation reinforces clarity, performance, and consistency with modern web development practices.

forEach

The forEach directive in Modular Cube enables declarative iteration over collections within HTML templates. It renders a repeated structure for each item in a given array or iterable, following a clear and scoped expression pattern. This approach brings structured, readable list rendering to markup using standard JavaScript expression syntax. This directive is designed to integrate seamlessly with the component's data context and maintain clarity in both simple and nested list structures.

üìå Feature supports

  • forEach directive must use the syntax: let item in items.
  • You can loop inline using array literals: let item of [1, 2, 3].
  • The value after in must be an array or iterable.
  • You cannot use multiple structural directives (if, forEach, etc.) on the same element.
  • You can nest forEach with *if or another forEach block.

üîç Syntax & Supported Contexts

<div forEach="$ {item} in $ {this.items}">
Content using item
</div>

  • $ {item} defines the loop variable for each iteration.
  • $ {this.items} must evaluate to an array or iterable object in the current context.
  • The loop variable is scoped to its instance within the repeated block.

üí° Notes

  • forEach creates a new scope for each iteration. The loop variable (e.g. item) is accessible inside that block only.

The forEach directive offers a concise and powerful way to generate HTML content dynamically based on collection data. With native expression support and clear scope handling, it supports structured iteration at any depth and promotes maintainable template design. Repeated content is easy to define and aligns with modern rendering practices through direct integration with JavaScript evaluation and native web capabilities.

Event Bindings

Supports binding JavaScript event handlers directly in your HTML using a simple and consistent syntax. You can handle DOM events like click, input, keydown, mouseenter, and more. This allows you to keep your event logic declarative and close to your markup. These bindings follow standard JavaScript event behavior but are declared directly in the template, no need for addEventListener or inline onclick attributes.

üìå Important Rules

  • Use the format: placeholderEvent="this.functionCall(...)".
  • The event name must be lowercase and match a standard DOM event (e.g. click, input, keydown).
  • You must append Event to the event name: clickEvent, inputEvent, keydownEvent, etc.
  • The value must be a valid JavaScript function call (with or without arguments).
  • The $event object can be passed to access native event data.

üí° Notes

  • ‚úÖ You can bind any number of events to an element.
  • ‚úÖ Any function accessible in the component‚Äôs scope can be used.
  • ‚úÖ $event gives you access to native event details (like target, key, etc.).
  • ‚ùå Do not use onClick, onInput, etc. ‚Äî use clickEvent, inputEvent, etc. instead.
  • ‚úÖ Event bindings can be combined with *if and forEach for dynamic UI behavior.

Event bindings enable clear, declarative interaction handling directly in markup using a consistent syntax aligned with native web standards. With scoped evaluation, support for $event, and flexible integration with other directives, this feature provides a practical and intuitive way to connect UI behavior to application logic.

Work with CSS

Variable

Template interpolation enables the embedding of JavaScript expressions directly within CSS content, using the $ {...} syntax. This works similarly to HTML template interpolation, allowing you to bind JavaScript properties or return values directly into your styles at render time.

üìå Important Rules

  • Expressions must be valid JavaScript and scoped to the current component (i.e. this.property or this.method()).
  • Only values that result in a valid CSS value are allowed (e.g., colors, units, keywords).
  • Function calls must return a value ‚Äî and must be pure (no side effects).
  • You can interpolate in inline styles, style blocks, or custom properties (e.g., --my-color: $ {this.color}).
  • Do not use interpolation inside CSS selectors or media queries ‚Äî only in property values.
  • Keep expressions simple and readable ‚Äî avoid chaining or complex logic inside styles.

üí° Notes

  • CSS interpolation works at render time ‚Äî changes to reactive properties will automatically reflect in styles.
  • This feature is framework-specific and not standard browser behavior. It is optimized for Shadow DOM scoping and dynamic styling needs.
  • Only string-compatible values will work ‚Äî e.g., no objects, arrays, or DOM references.
  • You can use component-scoped CSS variables for theming or design tokens, e.g., --theme-accent: $ {this.accentColor}.

Work with JS

Observable

The framework includes a lightweight Observable utility for managing event-based communication between components or logic blocks. This pattern allows you to decouple parts of your application by subscribing to and publishing named events, similar to a pub/sub model.

üìå Core Concepts

  • subscribe(name, subscriber): Register a function to run when a named event is published.
  • unsubscribe(name): Remove a previously registered event handler.
  • publish(name, payload): Trigger all subscriber callbacks for the given event name, passing in an optional payload.
  • Each event supports a single active subscriber ‚Äî subsequent calls with the same name will replace previous ones.

üí° Notes

  • Component A notifies Component B to refresh data.
  • This is a global system, use unique event names to avoid collisions.

The Observable system can be used for reactive data handling, custom event broadcasting, or integrating cross-component behaviors in a clean, manageable way.
